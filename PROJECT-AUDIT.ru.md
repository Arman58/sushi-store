# East West Delivery — Полный аудит проекта
**Дата:** 10.07.2026 · **Объём:** весь репозиторий (391 файл в src, ~52 265 строк) · **Стек:** Next.js 16 App Router, React 19, MUI 7, Prisma 6 / PostgreSQL, next-auth 4, next-intl, Zustand, Serwist PWA, Upstash Redis, Vercel

---

## 0. Резюме — сначала жёсткая правда

**Это не маркетплейс. Это магазин доставки еды одного продавца.** В `prisma/schema.prisma` нет моделей `Restaurant`/`Vendor`, нет курьеров, нет ничего мультитенантного. Любое сравнение с Uber Eats/DoorDash/Wolt рассыпается на уровне модели данных, а не UI. Если «маркетплейс» — реальная цель, это переписывание архитектуры, а не рефакторинг. Если же (что вероятнее и совершенно нормально) это один ресторан в Нор Ачне, то правильный ориентир — топовый D2C-магазин уровня Domino's или витрины *мерчанта* Wolt, и на этом фоне проект **на удивление хорош**.

**Вторая жёсткая правда: онлайн-оплаты нет.** `PaymentMethod` — это `CASH | CARD`, то есть карта *при получении*. Нет интеграции Stripe/Idram/ARCA/Telcell, нет payment intents, возвратов, вебхуков. Для доставки еды в 2026 году это главная недостающая функция с точки зрения выручки.

**Третья: инженерный фундамент сильно выше среднего** — серверная перепроверка цен, атомарное списание промокодов, CSP-nonce на каждый запрос, CSRF-middleware, отзыв JWT через Redis-denylist, cache tags, streaming Suspense, PWA, трёхъязычная i18n с таблицами переводов, CI с e2e на реальном Postgres. В большинстве «доставок еды», которые я аудирую, нет ничего из этого. Отдаю должное.

**Четвёртая: операционная гигиена — то, где всё разваливается.** В build-скрипте зашит хак с откатом миграции, история git — `fix / fix / fix prod`, `.idea/` и отладочные скрипты закоммичены, тестов мало (7 unit-файлов + 7 e2e-тестов на 52k строк), и на всех — один общий админский пароль.

**Итог: 6.5/10.** Крепкий магазин одного продавца с enterprise-инстинктами по безопасности, которого тянут вниз дисциплина эксплуатации, отсутствие платежей, отсутствие RBAC и допущения по масштабированию, которые ломаются после ~1–2 тыс. товаров.

## 1. Архитектура — 7/10

**Структура (8/10).** Feature-Sliced-Design-lite: `entities/ → features/ → widgets/ → shared/` плюс `app/`, `lib/`, `server/`. Правильный инстинкт, но соблюдается непоследовательно:

- `src/shared/ui/` замусорен бизнес-компонентами: `cart-drawer.tsx`, `cart-drawer-promo.tsx`, `min-order-progress.tsx`, `header-eta.tsx`, `empty-cart.tsx`. «Shared» — значит независимый от домена. UI корзины должен жить в `features/cart/ui/`. Сейчас направление зависимостей shared→features инвертировано (shared импортирует стор корзины), что убивает весь смысл слоёв.
- В `src/server/services/` лежит ровно **один** файл (`order.service.ts`), при этом 60+ серверных модулей живут в `src/lib/`. Две конкурирующие конвенции. Хуже того, существует ещё и `src/lib/order-service.ts`, который делает другое (обновление статуса/ETA). Два файла с именем «order service» в разных слоях — ловушка для сопровождения. Выберите один дом (`src/server/`), перенесите туда весь серверный код и добавьте защиту `import "server-only"`.
- `src/lib/` — свалка из 60 файлов, где смешаны server-only (`prisma.ts`, `otp-store.ts`), безопасные для клиента (`phone.ts`) и edge-only (`csp.ts`) модули, и ничто их не маркирует. Один неверный импорт из клиентского компонента — и вы отгружаете код рядом с секретами или ломаете edge-бандл.

**Границы Server/Client (7/10).** Использование RSC действительно хорошее — `src/app/[locale]/(store)/page.tsx` стримит секции через `Suspense` с продуманными по CLS скелетонами и держит LCP-hero вне Suspense (правильно, и комментарии в коде показывают, что автор понимает, почему). Но 149 из 391 файлов — `"use client"` (~38%). Главный виновник — MUI/Emotion (см. §3), а не плохая дисциплина.

**Управление состоянием (8/10).** Zustand-стор корзины (`src/features/cart/model/store.ts`) — одна из лучших реализаций корзины, что я видел: производный `qtyByProductId` + предвычисленные суммы, чтобы не делать O(n)-reduce в шапке, отложенная запись в localStorage ради INP, версионированная персистенция с миграциями v3→v5, троттлинг добавления по строке. Замечания — ниже в §2.

**Дизайн API (6/10).** REST-образные route handlers, более-менее консистентные. Проблемы: нет версионирования API, нет единого конверта ответа (часть роутов возвращает `{error}`, часть — plain text: `src/app/api/admin/login/route.ts` отдаёт ошибки `text/plain`, остальные — JSON), `src/app/api/admin/products/[id]/route.ts` — route handler на 700 строк с валидацией, переводами и диффом модификаторов внутри. Это сервис, а не роут.

**Маркеры техдолга.** В корне закоммичены отладочные артефакты `test-menu-query.ts` и `migration_diff.sql`. `.idea/` отслеживается git несмотря на запись в `.gitignore` (файлы добавили до правила — `git rm -r --cached .idea test-menu-query.ts migration_diff.sql`).

**Масштабируемость (4.5/10).** См. §3 (загрузка всего меню) и §16 (схема одного продавца). Текущий дизайн упирается в потолок: один ресторан, ~1–2 тыс. товаров, один город. Нормально, если это и есть бизнес; фатально, если в планах «маркетплейс».

## 2. Качество кода — 7.5/10

**Хорошее:** `strict: true`, всего 2 каста `as any` на 52k строк, zod-схемы для API-пейлоадов, дискриминированные коды ошибок (`OrderServiceError`, `UpdateOrderStatusError`), использование `satisfies`, исчерпывающе типизированные бакеты rate-limit. Это верхний дециль дисциплины типизации.

**Находки по убыванию серьёзности:**

1. **Захардкоженная русская строка в трёхъязычном приложении** — `src/features/cart/model/store.ts:324`: `appToastMessage: "Цены в корзине обновлены"`. Армянские и английские пользователи получают русский тост при ресинке цен корзины. Стор не может вызвать `useTranslations`, поэтому он должен эмитить *ключ* сообщения, а переводить его должен компонент снекбара (`src/shared/ui/layout-toast-snackbar.tsx`).

2. **`syncPricesWithServer` портит `basePrice` при изменении цены модификатора** — `store.ts:307-313` пересчитывает `basePrice = serverUnitPrice − Σ(устаревшие локальные дельты модификаторов)`. Если на сервере изменилась дельта *модификатора* (а не базовая цена), локальные дельты устарели и `basePrice` тихо становится неверным; следующее добавление того же товара с другими модификаторами даст неправильную цену строки. Сервер должен возвращать `basePrice` и дельты по каждому модификатору явно; никогда не выводить их вычитанием.

3. **Файлы-монстры.** `src/app/admin/(dashboard)/products/product-form-dialog.tsx` (842 строки), `orders/order-row.tsx` (776), `dashboard/page.tsx` (771), `orders/page.tsx` (767), `api/admin/products/[id]/route.ts` (700), `widgets/reviews/ui/product-reviews-section.tsx` (811). Всё, что больше ~300 строк, в этом коде коррелирует со смешением ответственности (состояние формы + API-вызовы + редактирование переводов + загрузка изображений в одном диалоге). Разделяйте по ответственности, выносите хуки.

4. **`src/lib/i18n-utils.ts:78,155` — единственные два `as any`, и они сидят в сердце локализации**, в функции, через которую проходит каждый запрос витрины. Типизируйте маппинг локализованных сущностей дженериками (`<T extends {translations: Tr[]}>`) вместо каста; переименование поля в схеме сейчас упадёт в рантайме, а не на компиляции.

5. **Дублированная логика подготовки данных.** Запрос «категории + первая картинка товара» повторяется в `menu-catalog-section.tsx`, `test-menu-query.ts` и (в вариациях) в виджетах главной. Вынесите один модуль `getStorefrontMenu(locale)`.

6. **35 прямых вызовов `console.log/error/warn`** параллельно с Sentry. Ошибки роутов должны идти через один логгер с контекстом Sentry; `console.error("[PUSH ERROR]", ...)` в `src/lib/order-service.ts:102` теряет контекст заказа.

7. **`decrementFirstLineForProduct`** (`store.ts:269`) уменьшает *произвольную* строку, когда у товара в корзине несколько вариантов модификаторов — с карточки товара пользователь не видит, какой вариант удаляет. UX-баг, зашитый в API стора. Кнопка «минус» на карточке при >1 строке должна открывать корзину/выбор варианта (Uber Eats делает ровно так).

## 3. Производительность — 6.5/10

**Что уже правильно:** streaming SSR с посекционным Suspense, `revalidate = 60` + `unstable_cache` с тегами + `revalidateTag` на админских записях (правильный ISR с инвалидацией), LCP-осознанный hero, отложенные записи в localStorage, ленивые модификаторы (`/api/menu/modifiers` по требованию вместо отгрузки всех модификаторов в пейлоаде меню — отлично), `use-progressive-render.ts` + IntersectionObserver для длинных списков, immutable-кэш для шрифтов/картинок.

**Проблемы:**

1. **🔴 Меню грузит все активные товары одним запросом без пагинации** (`menu-catalog-section.tsx`: `prisma.product.findMany({ where: { isActive: true }, include: { translations, category.translations, ... } })`). Ни `take`, ни курсора. На 200 товарах — нормально; на 2 000 — это многомегабайтный RSC-пейлоад и медленный TTFB на каждом промахе 60-секундного кэша. Тот же паттерн в поиске. Нужна пагинация в рамках категории (курсор по `position, id`) и виртуализированный рендер до того, как каталог вырастет.

2. **🔴 MUI + Emotion — ваш главный структурный налог на производительность.** Emotion — это runtime CSS-in-JS: он не работает в Server Components (именно *поэтому* 38% файлов клиентские) и пересериализует стили в рантайме (цена по INP на слабых Android — вашем основном рынке). Деревья, тяжёлые от `sx` (например, `theme.ts` на 715 строк), пересчитываются при обращении к теме. Миграция на Pigment CSS (zero-runtime преемник MUI, совместим с `sx`) или как минимум статическая экстракция через `@mui/material-nextjs` должна быть в роадмапе. Это разница между «ощущается как нативное приложение» и «ощущается как сайт» на телефоне за $150.

3. **Полный бандл локали уезжает каждому клиенту.** `src/app/[locale]/layout.tsx` делает `getMessages()` и передаёт *все* сообщения (`hy.json` = 75 КБ) в `NextIntlClientProvider`. Строки админки, писем, чекаута — на главной странице. next-intl умеет выбирать неймспейсы: `messages={pick(messages, ['common','cart','menu'])}` на сегмент layout. Лёгкая экономия ~40–50 КБ на первой загрузке.

4. **Три библиотеки анимаций/каруселей.** framer-motion (~30 КБ gz) + swiper (~40 КБ) + переходы MUI. Swiper ради одной промо-карусели — дорого; CSS scroll-snap закрывает 90%. framer-motion для переходов страниц (`template.tsx`) — проверьте, не заменит ли его уже View Transitions API.

5. **Нет бюджета бандла.** Добавьте `@next/bundle-analyzer` и проверку размера в CI. Нельзя управлять тем, что не измеряешь; собрать проект в этой песочнице я не мог (нет DATABASE_URL), так что первая задача после аудита: `ANALYZE=true next build`.

6. **Индексы БД:** у `OrderItem` **нет индекса по `orderId`** — Prisma/Postgres НЕ индексирует FK-колонки автоматически. Каждый include `order.items` делает seq scan; будет больно после ~100k позиций заказов. Добавьте `@@index([orderId])`. Также не хватает: `Product @@index([categoryId, isActive, position])` для страниц категорий. У `Order` всё в порядке (`status,createdAt` покрыт).

7. **Почти-N+1:** `deliveryZone.findFirst({ include: { translations: true } })` и поштучные lookup'ы зон на этом масштабе терпимы, но `prepareOrderItems` + проверка промо + проверка зоны — последовательные await'ы в `createOrder`: ~3 последовательных обращения к БД до транзакции. Распараллельте чтения.

**Core Web Vitals:** архитектура обещает хорошие LCP/CLS (hero в первичном HTML, зарезервированные aspect-ratio, комментарии показывают понимание CLS). Риск — INP (Emotion + MUI на средних мобильных). Подключите `useReportWebVitals` → Vercel Analytics (установлен — проверьте, что кастомные vitals реально идут) и задайте бюджеты: LCP < 2.5s p75 mobile, INP < 200ms.

## 4. UX-аудит — 7/10

Оценка по структуре кода (кликнуть работающий инстанс я не могу; делайте на это скидку).

**Есть и сделано хорошо:** гостевой чекаут с токенизированным трекингом заказа (`Order.accessToken` + `/order/[id]?key=`), живой статус заказа по SSE (`/api/order-status/stream`), слоты предзаказа с серверной валидацией, гейтинг по часам работы с фолбэком на предзаказ, «с какой суммы готовить сдачу» (`changeFrom` + `CHANGE_FROM_TOO_SMALL` — деталь, о которой забывают почти все), сохранённые адреса, прогресс-бар до минимального заказа (`min-order-progress.tsx` — отличный конверсионный паттерн), промо-дровер для новых посетителей, pull-to-refresh, ресинк цен корзины с видимым тостом, повалидационная проверка строк корзины (`use-cart-line-validation.ts`), скелетоны + `loading.tsx` на всех основных роутах, состояния ошибок/пустоты (`empty-cart.tsx`, `fatal-error-screen.tsx`, `menu-load-error.tsx`), хаптика (`haptic.ts`).

**Разрывы с классом Uber Eats/Wolt:**

1. **Клиент не может отменить заказ.** Поток статусов управляется только кухней. Стандарт индустрии: отмена в течение N минут, пока статус NEW.
2. **Нет живого трекинга курьера / карты.** ETA — это таймстемп, который выставляет кухня (`estimatedDeliveryAt`). Для одного ресторана нормально, но даже статичная карта-прогресс «готовим → в пути» поднимает воспринимаемое качество.
3. **Зона доставки — выпадающий список** (`DeliveryZone`), а не определение по адресу. Wolt/Glovo резолвят зону геокодингом адреса. Трение + источник ошибок («выбрал не ту зону» → `requiresManagerApproval` как костыль).
4. **Неоднозначность `decrementFirstLineForProduct`** (§2.7).
5. **Избранное живёт только в localStorage** (ключ `ew_favorites`, `src/features/favorites/model/store.ts`) — теряется при смене устройства, невидимо между устройствами залогиненного пользователя, бесполезно для персонализации. Паттерн серверной синхронизации `Cart` уже есть — отзеркальте его для избранного.
6. **Трение в авторизации:** email + пароль + OTP-верификация. Для доставки еды phone-first вход по OTP (телефон и OTP-инфраструктура уже есть) конвертит сильно лучше; email/пароль — паттерн 2015 года. Соцлогина тоже нет.
7. **Нет петли запроса отзыва после заказа** — отзывы есть (флаг verified purchase, голоса «полезно»: реально хорошо), но в order-success/трекере, судя по коду, ничего их не запрашивает.
8. Чекаут — один хук формы на 389 строк (`useCheckoutForm.ts`) + DeliverySection на 587 строк — проверьте на устройстве, что он подаётся как перевариваемые шаги (контакты → доставка → оплата), а не одна длинная форма; длинные формы измеримо роняют мобильную конверсию.

## 5. Mobile-First — 7.5/10

Сильно: отдельные `mobile-bottom-nav.tsx`, `mobile-nav-drawer.tsx`, паттерны bottom-sheet, `use-visual-viewport-bottom-inset.ts` (инсеты с учётом клавиатуры — редкая внимательность), `use-scroll-hide.ts`, `pull-to-refresh.tsx`, хаптика, `pwa-nav-arrows.tsx` для standalone-режима, `InstallPwaPrompt.tsx`, офлайн-страница (`/offline` корректно исключена из i18n-роутинга — комментарий показывает, что падение установки SW реально дебажили), push-уведомления end-to-end (VAPID, подписки на пользователя, локализованные пуши).

Пробелы: нет проверки `screenshots`/rich install UI в `manifest.ts` (проверьте), нет background sync для заказов, оформленных офлайн (очередь-и-ретрай через Serwist — следующий шаг PWA), обработку safe-area надо проверить на устройствах с вырезом (использования `env(safe-area-inset)` в просмотренных файлах я не нашёл), и PWA не заменяет присутствие в сторах, если важен рост (позже — обёртка Capacitor, не сейчас).

## 6. UI — 7/10 (по коду; визуальную полировку отсюда не проверить)

Один файл темы (`src/shared/ui/theme.ts`, 715 строк) со светлой/тёмной темой, защита от FOUC через `theme-init-script.tsx` + предпочтение в cookie (`theme-preference.server.ts`) — корректная SSR-реализация тёмной темы, которую большинство команд делает неправильно. Единые обёртки `AppButton/AppInput/AppSelect` = зарождающаяся дизайн-система; продолжайте прятать все сырые импорты MUI за этими обёртками, чтобы будущий уход с MUI был возможен. `skeleton-styles.ts` централизует loading UI.

Проблемы: монолитную тему на 715 строк стоит разделить на токены (палитра/типографика/отступы) и переопределения компонентов; дизайн-токены заперты в теме MUI, а не в CSS-переменных (MUI 7 поддерживает `cssVariables: true` — тема без стоимости гидрации); нет Storybook — у «библиотеки компонентов» нет ни контракта, ни визуальной регрессионной сетки.

## 7. Продуктовый аудит — 5.5/10

**Есть:** категории, модификаторы с required/max, апселлы («с этим берут» через `ProductUpsell`), отзывы + рейтинги + verified purchase + голоса «полезно», промокоды с JSON conditions/actions (бесплатная доставка, скидка на категорию), зоны доставки с минимальным заказом, доставка ко времени, самовывоз, гостевой чекаут, история заказов, сохранённые адреса, недавно просмотренные, избранное (локально), пуши, три языка (hy/ru/en), Telegram-бот кухни с inline-кнопками статусов (отличный операционный инструмент для маленького ресторана).

**Не хватает, по влиянию на выручку для магазина одного продавца:**

1. **Онлайн-оплата** (Idram/ARCA/Telcell для Армении + Stripe для карт). Наложенный платёж ограничивает средний чек и порождает потери от «не забрал».
2. **Повторный заказ в один тап** из истории (данные есть; проверьте, что UI это показывает — фича с максимальным ROI в доставке еды, ~30% заказов у лидеров).
3. **Серверное избранное + персонализация «заказать снова» на главной.**
4. **Комбо/сеты** — модели комбо в схеме нет; суши — категория, живущая сетами, это реальный пробел.
5. **Лояльность/кэшбек** (хотя бы простой счётчик баллов) и **реферальные коды** — инфраструктура `PromoCode` покрывает 70% пути.
6. **Фото в отзывах** (у `Review` нет поля images) — фотоотзывы измеримо поднимают конверсию на еде.
7. **Поиск:** `/api/search` есть с rate limit; нет подсказок/толерантности к опечаткам. При таком размере каталога клиентский fuzzy (fuse.js по закэшированному меню) бьёт серверный round-trip.
8. **Геокодинг адреса + автоопределение зоны** (см. §4.3).
9. Подарочные карты, мультивалютность — пропустить: не тот рынок/стадия. Голосовой/AI-поиск — пропустить: игрушка на этом масштабе.

## 8. Админ-панель — 6/10

**Не хватает для «enterprise»:**

3. **Нет массовых операций** (массовое обновление цен, массовое включение/выключение) и нет импорта/экспорта товаров — больно при обновлении меню.
4. **Нет склада/остатков** кроме булевого `isAvailable` — ни «нет до завтра» с автосбросом, ни количеств.
5. Фиче-флаги: `LAUNCHDARKLY_ONBOARDING.md` — *брошенная на середине* интеграция (шаг 4 из 6, SDK «not started»). Либо доделайте, либо удалите документ; полуустановленный вендорский онбординг в репозитории — шум. Для команды такого размера таблица `FeatureFlag` + тумблер в админке лучше контракта с LaunchDarkly.

## 10. Безопасность — 8/10 (лучшая область проекта)

Есть и сделано правильно: CSP-nonce на каждый запрос в middleware с enforce на проде / report-only в dev и защитой от отравления CDN-кэша (`proxy.ts:38-43` — автор читал CVE), same-origin CSRF-проверка всех мутирующих `/api` с явным списком исключений, `timingSafeStringEqual` на логине админа, админский JWT с jti + denylist в Redis, префиксы кук `__Host-`/`__Secure-`, HSTS + XFO + nosniff + Permissions-Policy, бакеты rate-limit по классам эндпоинтов, zod-валидация + `parse-json-body`, валидация загрузок (размер, тип, только для админа, Cloudinary), OTP-хранилище, падающее *закрыто* на проде без Redis, bcrypt, `escape-html.ts` для писем.

**Находки:**

1. **🟠 Rate limit на `adminLogin` в проде падает ОТКРЫТО, когда Upstash недоступен/не настроен** (`rate-limit.ts:109-116` — сознательная политика «не терять выручку», применённая ко *всем* бакетам). Для `order` — правильно; для `adminLogin`/`verifyOtp`/`register` — нет. Сбой Redis = нелимитированный брутфорс единственного общего админского пароля. Сделайте auth-бакеты fail-closed (или фолбэк на in-memory лимитер на инстанс).
2. **🟠 Общий админский креденшл + отсутствие RBAC** (§8.1) — крупнейший реальный риск: переиспользование/утечка пароля → полный админ. Добавьте персональные админ-аккаунты и TOTP 2FA на вход.
3. **🟡 `getClientIp` доверяет левому значению `x-forwarded-for`** (`rate-limit.ts:82-84`). На Vercel заголовок контролируется платформой, но код переносим, и за любым другим reverse proxy клиент может подделывать XFF и бесконечно менять identity лимитера. Сначала берите `x-real-ip` (или `x-vercel-forwarded-for`) и задокументируйте допущение.
4. **🟡 Редирект после POST логина админа по умолчанию 307** (`api/admin/login/route.ts:43`), то есть POST (с креденшлами в теле) переотправляется на `/admin/orders`. Браузеры переживут, но семантика неверная и пароль ретранслируется на page-роут. Используйте `303`.
5. **🟡 Экранирование `</script>` в JSON-LD** (§9.2).
6. Fail-open на Redis при *отзыве* админ-сессий (`admin-session.ts:63-66`): отозванные сессии живут во время сбоя Redis. Приемлемый компромисс, но зафиксируйте в модели угроз — с TTL токена 12ч риск ограничен.
7. Хорошо: `.env*` корректно в gitignore и не отслеживается (проверено по `git ls-files`).

## 11. База данных — 7.5/10

Схема продуманная: деньги целыми AMD (задокументировано!), snapshot-колонки на `Order` (`deliveryZoneName`, `deliveryPrice`, снимки имени/цены в `OrderItem`) — история переживает правки каталога, таблицы переводов с `@@unique([entityId, locale])`, денормализованные `ratingAvg/ratingCount` + `helpfulCount` с внятными комментариями, каскады в основном правильные, атомарный инкремент `timesUsed` промокода сырым SQL с проверкой `maxUsages` внутри транзакции (`order.service.ts:169-184` — по учебнику).

Исправить:

1. **У `OrderItem` нет `@@index([orderId])`** (§3.6) — самая запрашиваемая связь в системе.
2. **Нет CHECK-ограничений:** `Review.rating` 1..5, `Product.price >= 0`, `OrderItem.quantity > 0` валидируются только в API. Prisma поддерживает сырые CHECK в миграциях; кривой скрипт или будущий баг админки молча запишет мусор.
3. **`PromoCode.conditions/actions` — нетипизированный `Json`** — семантика промо живёт наполовину в JSON БД, наполовину в `src/lib/promo.ts`. Версионируйте форму JSON (`{v:1, rules:[...]}`), иначе её никогда не мигрировать безопасно.
4. `Product.images Json` — то же: никакой гарантии формы; типизированная таблица `ProductImage` заодно даст alt-текст на изображение (SEO §9).
5. У `OrderItem.productId` намеренно нет FK («просто храним id») — оправданный snapshot-паттерн, но у `CartLine.productId` FK с каскадным удалением *есть*: **удаление товара молча выкидывает его из серверных корзин пользователей**, при этом копия в localStorage остаётся → гарантированные конфликты синка. Ставьте `onDelete: Restrict` + soft delete через `isActive=false`; флаг уже есть.
6. Артефакт SQLite `prisma/dev.db` рядом со схемой postgres — удалить; путает каждого нового разработчика.

## 14. Бизнес-логика — 8/10 (самая сильная инженерия в репозитории)

`src/server/services/order.service.ts` — украшение проекта: серверная перепроверка цены каждой позиции по БД (`prepareOrderItems`), сверка заявленного и проверенного на трёх уровнях (позиции, subtotal, скидка, итог → `PRICE_MISMATCH`/`SUBTOTAL_MISMATCH`/`DISCOUNT_MISMATCH`/`TOTAL_MISMATCH`), минимальный заказ по зоне, ревалидация промо на сервере с атомарным списанием внутри транзакции, валидация часов работы и слотов, безопасное к уникальности телефона обновление профиля в той же транзакции, валидация сдачи. Клиент не может соврать этому эндпоинту. Лучше, чем у многих профинансированных стартапов.

Оставшиеся кейсы:

1. **Промо считается по коду, а не по пользователю** — нет `maxUsagesPerUser`, один пользователь может применить код N раз в разных заказах. Добавьте уникальный трекинг `PromoRedemption(userId/phone, promoId)`.
2. **Проверка `TOTAL_MISMATCH` выполняется *внутри* транзакции после инкремента промо** (`order.service.ts:186`) — по откату корректно, но зря держит блокировку строки промо; вынесите выше `$transaction`.
3. **Телефон-заглушка `"-"` для самовывоза** (`order.service.ts:190-195`) — магическая строка-сентинел в БД. Сделайте `phone` nullable или валидируйте нормально; теперь каждый потребитель должен знать про `"-"`.
4. **Нет ключа идемпотентности на создании заказа.** Двойной тап + медленная сеть + ретрай = дубль заказа. Rate limit (5/мин) снижает, но не исключает. Принимайте заголовок `Idempotency-Key` с unique-constraint на 24ч.
5. Блокировка отменённых есть (`CANCELLED_LOCKED`), но конечного автомата статусов нет — любой статус прыгает в любой (DONE → COOKING). Закодируйте допустимые переходы.
6. Модели налогов/сборов нет вообще — для наличной Армении сегодня нормально, для карточных платежей завтра блокер (фискальные чеки).

**Провалы:**

3. **История git — `fix prod / fix prod / fix perfomance / fix / fix`.** Не бисектится, не ревьюится, нет гранулярности revert. Введите conventional commits; CI для их enforcement уже есть (commitlint).
4. **Отслеживаемый мусор:** `.idea/` (11 файлов), `test-menu-query.ts`, `migration_diff.sql` — всё закоммичено. `git rm --cached`; `.gitignore` их уже покрывает.
5. **Тестов мало:** 7 unit-файлов (все на денежных путях — приоритет верный) + 3 e2e-спека / 7 тестов на 52k строк. Ноль тестов на: движок JSON-условий промо, миграции стора корзины v3→v5 (миграции персистентного состояния — ровно то место, где случается тихая потеря данных), админские роуты, upsert отзывов, `syncPricesWithServer` (§2.2). Нет отчёта о покрытии.
6. Нет `docker-compose.yml` для локальных Postgres+Redis, нет `.nvmrc`, нет pre-commit хуков (husky + lint-staged).
